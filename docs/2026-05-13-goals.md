# 2026-05-13 Goals: macOS-first dotstate evolution

## Default choices to adopt

- **Product stance:** dotstate is the ergonomic, safety-focused macOS developer-state orchestrator over proven backends, not a replacement for every backend.
- **Primary target:** optimize the next architecture slice for macOS Tahoe on Apple Silicon first; keep Windows compatibility in mind, but do not let it block macOS module design.
- **File-state backend:** keep Chezmoi copy semantics as the default for managed files.
- **Package/app backends:** default to Homebrew Bundle for formulae/casks/taps, `mas` for Mac App Store inventory, and bundle identifiers for direct-download apps.
- **macOS state model:** use typed modules instead of expanding filesystem crawling: files, brew, mas, apps, defaults, launchd, profiles, privacy/TCC, subrepos, and secrets.
- **Safety model:** every module must support capability-aware facts and plan-first behavior before broad apply behavior.
- **Privacy/security model:** never capture secret values, Keychain contents, or TCC databases; store references, metadata, redacted facts, or manual checkpoints only.
- **Permission model:** treat Full Disk Access, sudo, SIP, MDM, and TCC as first-class capabilities with clear diagnostics.
- **Extensibility model:** prove first-party modules and versioned schemas before exposing public executable plugins.
- **Advanced mode:** keep nix-darwin/Home Manager optional; do not make Nix the default onboarding path.

## Contract baseline now in force

The module/state contract and macOS surface contract are specified in:

- [docs/specs/modules.md](specs/modules.md)
- [docs/specs/macos-state-surfaces.md](specs/macos-state-surfaces.md)

Hindsight from the contract pass:

- Upcoming implementation must use the `dotstate.fact.v1`, `dotstate.plan.v1`, `dotstate.result.v1`, `dotstate.diagnostic.v1`, `dotstate.backup.v1`, and `dotstate.audit.v1` envelopes instead of inventing per-module shapes.
- Add fixture cases before implementation for each surface touched; include redaction sentinel tests before any report or plan is considered share-safe.
- Keep `auto_apply` disabled until plan, backup/restore or explicit reversibility, idempotency, and permission-denied diagnostics are covered by tests.
- Treat absent desired-state artifacts as "no desired state declared", not as delete requests.
- Use diagnostics for missing tools, locked accounts, Full Disk Access gaps, sudo needs, MDM ownership, and unsupported surfaces so read-only audit can remain useful without elevation.

## Module orchestrator baseline now in force

`dot apply`, `dot capture`, and `dot sync` now route through the shared module orchestrator in `internal/modules`, with Chezmoi implemented as the first `files` module.

Hindsight from the orchestrator pass:

- New mutating modules should plug into the orchestrator rather than adding bespoke CLI paths.
- Module plans and results should keep using the shared `dotstate.plan.v1`, `dotstate.result.v1`, `dotstate.diagnostic.v1`, and `dotstate.backup.v1` records.
- `dot apply --dry-run`, `dot capture --dry-run`, and `dot sync --dry-run` are now the baseline UX for reviewing module plans. Future module commands should provide equivalent plan-first output before mutation.
- The files module backs up managed destination files before apply and verifies with a post-apply diff. Future modules need equivalent backup/restore or explicit reversibility before `auto_apply`.
- `dot sync` now refuses to start from a dirty repo and reports rebase conflicts with porcelain status. Future sync/capture work should preserve that guardrail instead of sweeping unrelated repo changes into generated commits.
- Dry-run-only or unsupported module changes are blocked by the orchestrator unless a module explicitly exposes `auto_apply`.

## Goal 1: Build read-only `dot macos audit --json`

**Outcome:** ship a non-mutating macOS audit command that reports normalized state facts using `dotstate.audit.v1`.

**Default MVP modules:**
1. Homebrew taps/formulae/casks and Brewfile presence.
2. Mac App Store app inventory via `mas` when available.
3. Installed `.app` bundles with bundle IDs and source hints.
4. User LaunchAgents and Homebrew services.
5. Curated `defaults` domains in read-only mode.
6. Configuration profiles and privacy/TCC posture as report/manual/MDM-only facts.

**Contract updates:**
- Emit facts and diagnostics according to [docs/specs/modules.md](specs/modules.md).
- Reuse the `internal/modules` schema types and module interfaces instead of adding an audit-only parallel abstraction.
- Follow the surface order, IDs, desired artifact conventions, and fixture matrix in [docs/specs/macos-state-surfaces.md](specs/macos-state-surfaces.md).
- Add golden fixtures for each implemented surface before depending on local-machine output.
- Do not prompt for sudo, request Full Disk Access, unlock accounts, or mutate desired-state artifacts during audit.

**Acceptance:** command exits successfully without elevated permissions, redacts sensitive values, labels unavailable tooling clearly, and emits stable JSON suitable for golden tests.

## Goal 2: Reduce discovery noise and make onboarding guided

**Outcome:** replace broad noisy discovery with high-signal, curated onboarding.

**Scope:**
- Prefer curated app recipes and typed module facts over deep `~/Library` crawling.
- Add user-maintained ignore/curation registries.
- Implement planned `discover --roots` and `--max-file-size` flags.
- Exclude caches, language-server payloads, package-manager vendor trees, source maps, browser databases, and generated app bundles by default.
- Add an interactive TUI that explains why each candidate is recommended, maybe, risky, ignored, or better handled by a typed module.
- Feed promising non-file findings into module capture workflows rather than expanding the file crawler.
- Preserve `dot sync` dirty-repo safeguards when discovery suggests repo mutations.

**Acceptance:** real macOS discovery produces a short actionable review list by default; deep scans remain explicit opt-in.

## Goal 3: Add macOS bootstrap and scheduling as first-class flows

**Outcome:** make a fresh Mac reproducible with minimal manual work and clear checkpoints.

**Scope:**
- Publish GitHub Releases binaries for Apple Silicon.
- Add a one-line bootstrap script that installs/downloads `dot`, checks Xcode Command Line Tools, and guides Homebrew setup.
- Make 1Password/op unlock a deliberate manual checkpoint.
- Add `dot schedule install|status|remove` for a user LaunchAgent.
- Keep shutdown flush best-effort and non-destructive.
- Ensure bootstrap can reach `dot apply --dry-run`, `dot sync --dry-run`, and `dot macos audit --json`, and interpret capability diagnostics without internal docs.

**Acceptance:** a clean Mac can reach `dot doctor`, `dot apply --dry-run`, and `dot macos audit --json` without reading internal docs.

## Goal 4: Capture and reconcile non-file state

**Outcome:** make capture bidirectional where safe and explicit where not safe.

**Scope:**
- Capture Homebrew state into a Brewfile-compatible artifact.
- Capture MAS app IDs when `mas` is available.
- Capture app bundle IDs/source hints for reviewable direct-download checkpoints.
- Capture curated defaults only; avoid full plist dumps by default.
- Implement subrepo clone-if-missing apply and `dot subrepo status`.
- Integrate secret-backed generated files without committing generated secret material.
- Use the desired artifact paths and absent-artifact semantics from the module contract.
- Plug mutating non-file modules into the existing orchestrator with dry-run, backup/restore or explicit reversibility, apply, and verify phases.

**Acceptance:** capture produces reviewable, portable artifacts and labels machine-specific or manual-only state instead of hiding it.

## Goal 5: Harden redaction, secret handling, and macOS privacy diagnostics

**Outcome:** make secret safety and macOS permission friction core product behavior.

**Scope:**
- Replace placeholder gitleaks behavior with parsed scanner output or an explicit unavailable diagnostic.
- Propagate sensitivity/taint through reports, plans, logs, errors, backups, and result records emitted by the module orchestrator.
- Keep Keychain as metadata/reference-only.
- Detect permission-denied cases and explain likely remedies: Full Disk Access, TCC prompt, sudo, SIP, or MDM restriction.
- Add tests proving reports never print secret values, including sentinel values in fixtures.

**Acceptance:** users can safely run reports and share sanitized output without leaking credentials.

## Goal 6: Expand verification around macOS behavior

**Outcome:** make advanced macOS behavior reproducible and regression-resistant.

**Scope:**
- Add the module fixture layout from [docs/specs/modules.md](specs/modules.md) under `test/fixtures/modules/v1/`.
- Add golden tests for facts, plans, redaction, path normalization, capability diagnostics, audit envelopes, and orchestrator result records.
- Add sandbox-home e2e tests for capture/apply/diff/backup/restore.
- Add macOS runner tests for Brewfile inventory, defaults reads, LaunchAgent detection, and permission-denied diagnostics.
- Add idempotency tests: apply twice equals no-op.
- Add interrupted-apply restore tests.

**Acceptance:** release readiness can be judged from repeatable artifacts under `state/e2e-runs/<timestamp>/`.

## Suggested sequence

1. **P0:** add the audit fixture harness and first audit/redaction sentinel tests on top of the shared module schemas.
2. **P1:** implement read-only `dot macos audit --json` with brew/apps/launchd/defaults/profile facts through the module interfaces.
3. **P2:** reduce discover noise and add guided TUI onboarding, reusing module facts where possible.
4. **P3:** add bootstrap releases and LaunchAgent scheduling that can explain module dry-run/audit diagnostics.
5. **P4:** add capture/apply modules for Brewfile/MAS/apps/defaults/subrepos using the existing orchestrator backup/verify gates.
6. **P5:** consider public plugin APIs only after first-party modules pass contract tests.

## Non-goals for the next slice

- Do not build a bigger generic filesystem crawler as the primary macOS strategy.
- Do not make Nix mandatory for normal onboarding.
- Do not mutate TCC/privacy databases or Keychain contents.
- Do not expose third-party executable plugins before schemas, redaction, and contract tests are stable.
- Do not apply macOS defaults broadly from full plist exports.
