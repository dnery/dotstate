# 2026-05-13 Goals: macOS-first dotstate evolution

## Default choices to adopt

- **Product stance:** dotstate is the ergonomic, safety-focused macOS developer-state orchestrator over proven backends, not a replacement for every backend.
- **Primary target:** optimize the next architecture slice for macOS Tahoe on Apple Silicon first; keep Windows compatibility in mind, but do not let it block macOS module design.
- **File-state backend:** keep Chezmoi copy semantics as the default for managed files.
- **Package/app backends:** default to Homebrew Bundle for formulae/casks/taps, `mas` for Mac App Store inventory, and bundle identifiers for direct-download apps.
- **macOS state model:** use typed modules instead of expanding filesystem crawling: files, brew, mas, apps, defaults, launchd, profiles, privacy/TCC, subrepos, and secrets.
- **Safety model:** every module must support capability-aware facts and plan-first behavior before broad apply behavior.
- **Privacy/security model:** never capture secret values, Keychain contents, or TCC databases; store references, metadata, redacted facts, or manual checkpoints only.
- **Permission model:** treat Full Disk Access, sudo, SIP, MDM, and TCC as first-class capabilities with clear diagnostics.
- **Extensibility model:** prove first-party modules and versioned schemas before exposing public executable plugins.
- **Advanced mode:** keep nix-darwin/Home Manager optional; do not make Nix the default onboarding path.

## Contract baseline now in force

The module/state contract and macOS surface contract are specified in:

- [docs/specs/modules.md](specs/modules.md)
- [docs/specs/macos-state-surfaces.md](specs/macos-state-surfaces.md)

Hindsight from the contract pass:

- Upcoming implementation must use the `dotstate.fact.v1`, `dotstate.plan.v1`, `dotstate.result.v1`, `dotstate.diagnostic.v1`, `dotstate.backup.v1`, and `dotstate.audit.v1` envelopes instead of inventing per-module shapes.
- Add fixture cases before implementation for each surface touched; include redaction sentinel tests before any report or plan is considered share-safe.
- Keep `auto_apply` disabled until plan, backup/restore or explicit reversibility, idempotency, and permission-denied diagnostics are covered by tests.
- Treat absent desired-state artifacts as "no desired state declared", not as delete requests.
- Use diagnostics for missing tools, locked accounts, Full Disk Access gaps, sudo needs, MDM ownership, and unsupported surfaces so read-only audit can remain useful without elevation.

## Module orchestrator baseline now in force

`dot apply`, `dot capture`, and `dot sync` now route through the shared module orchestrator in `internal/modules`, with Chezmoi implemented as the first `files` module.

Hindsight from the orchestrator pass:

- New mutating modules should plug into the orchestrator rather than adding bespoke CLI paths.
- Module plans and results should keep using the shared `dotstate.plan.v1`, `dotstate.result.v1`, `dotstate.diagnostic.v1`, and `dotstate.backup.v1` records.
- `dot apply --dry-run`, `dot capture --dry-run`, and `dot sync --dry-run` are now the baseline UX for reviewing module plans. Future module commands should provide equivalent plan-first output before mutation.
- The files module backs up managed destination files before apply and verifies with a post-apply diff. Future modules need equivalent backup/restore or explicit reversibility before `auto_apply`.
- `dot sync` now refuses to start from a dirty repo and reports rebase conflicts with porcelain status. Future sync/capture work should preserve that guardrail instead of sweeping unrelated repo changes into generated commits.
- Dry-run-only or unsupported module changes are blocked by the orchestrator unless a module explicitly exposes `auto_apply`.

## Bootstrap and scheduling baseline now in force

macOS bootstrap now has a first-class path: GitHub Release assets for `dot-darwin-arm64`, `scripts/bootstrap-macos.sh`, explicit Xcode/Homebrew/1Password checkpoints, a bootstrap-safe `dot macos audit --json` envelope, and `dot schedule install|status|remove` for the user LaunchAgent.

Hindsight from the bootstrap/scheduling pass:

- The bootstrap script consumes the `dot-darwin-arm64.tar.gz` release asset and installs into `~/.local/bin` by default; future release work must preserve asset names or update the script in the same change.
- `dot macos audit --json` preserves the bootstrap command shape and now emits `dotstate.audit.v1` facts plus capability diagnostics. Future audit changes should extend the same command rather than introduce a second shape.
- `dot schedule install` renders `~/Library/LaunchAgents/com.dnery.dotstate.sync.plist` and runs `dot --config <repo>/dot.toml sync` on the configured interval. Future scheduling changes must keep `--dry-run` review and dotstate-owned plist protections.
- macOS shutdown flush remains intentionally manual/best-effort. Do not add root hooks, logout hooks, or shutdown-time apply behavior without a separate safety design.
- 1Password/op remains a manual checkpoint during bootstrap. Bootstrap must never fetch or print decrypted secret values.

## Redaction and privacy diagnostics baseline now in force

Secret safety now has shared implementation instead of ad hoc output checks: `internal/redact` sanitizes secret-looking strings, module plans/run reports are sanitized before return, runner errors and structured logs redact sensitive values, discover reports parse/redact gitleaks JSON when invoked, and `dot discover --report` emits `secrets.gitleaks.unavailable` when the external scanner is missing.

Hindsight from the redaction/privacy pass:

- Future modules should rely on the shared sanitizer and sensitivity promotion before returning facts, plans, backups, results, or diagnostics. Do not add one-off string filtering in new modules.
- Sentinel tests using `DOTSTATE_TEST_SECRET_DO_NOT_PRINT` now cover discover reports, module plans/results/backups, runner errors, logs, bootstrap output, schedule plans, and audit JSON. Keep adding sentinel cases at each new output boundary.
- `dot macos audit --json` now includes bootstrap guardrails for Full Disk Access, TCC, SIP, MDM, and Keychain reference-only behavior. Future audit collectors should refine these diagnostics with real facts without reading TCC databases or Keychain contents.
- Backup records can be tainted as `secret` while local payloads remain restorable and unprinted. Future restore UX must preserve that distinction.
- Missing gitleaks is a diagnostic, not a hard failure. Chezmoi `--secrets=error` remains the final add guardrail.

## Verification baseline now in force

Repeatable verification now has a concrete fixture and harness baseline: `test/fixtures/modules/v1/` contains first files-module and bootstrap-audit golden cases, `make test-e2e-verify` produces a `macos-verification` bundle, and generated review artifacts are scanned by `test/e2e/verify_artifacts_no_sentinel.sh`.

Hindsight from the verification pass:

- Future module work should add or extend fixture cases first, then update goldens with `DOTSTATE_UPDATE_GOLDEN=1 go test ./internal/modules ./internal/macos` only after reviewing the diff.
- Every new output boundary should include `redaction.assert_absent.txt` coverage and pass the e2e sentinel verifier before being considered share-safe.
- The verification harness now exercises files apply dry-run, apply, backup payload creation, idempotent second apply, capture dry-run, capture, schedule install/status/remove in a sandbox, and release bootstrap dry-run. Future e2e scenarios should add checks to this bundle instead of creating unrelated artifact shapes.
- Read-only macOS command probes for `brew bundle check`, curated `defaults read`, and user `launchctl print` lock in safe runner command shapes for the upcoming audit collectors.
- interrupted-apply restore coverage proves secret-tainted backup records can remain share-safe while local payloads stay restorable; future mutating modules need equivalent restore tests before `auto_apply`.

## Read-only audit baseline now in force

`dot macos audit --json` now emits real `dotstate.audit.v1` facts through the module audit interface instead of pending bootstrap diagnostics. The command remains non-mutating and does not request sudo, Full Disk Access, Keychain unlocks, or TCC database reads.

Hindsight from the read-only audit pass:

- Remaining discovery and capture work should consume the typed audit facts before falling back to broad filesystem crawling.
- Homebrew, MAS, apps, launchd, defaults, profiles, privacy/TCC, subrepos, and secrets now have read-only fact or diagnostic shapes that future capture/apply modules should preserve.
- Missing tools, unsigned-in accounts, unreadable plists, absent curated defaults, privacy restrictions, and MDM/report-only posture are diagnostics, not hard failures.
- The golden fixture `test/fixtures/modules/v1/macos/audit-readonly/` is the regression anchor for the current audit envelope. Extend it when adding surfaces or fields.
- Audit uses only read-only command shapes such as `brew tap`, `brew list --formula/--cask --versions`, `brew bundle check --file`, `mas list`, `plutil -convert json -o -`, `launchctl print`, `brew services list --json`, `defaults read <domain> <key>`, and `profiles status -type enrollment`.

## Discovery/onboarding baseline now in force

`dot discover` now defaults to curated, high-signal file roots and treats broad filesystem crawling as explicit opt-in through `--deep` or `--roots`. It explains typed-module handoff points instead of trying to discover macOS apps, Homebrew, MAS, launchd, defaults, profiles, privacy/TCC, subrepos, or secrets by crawling `~/Library`.

Hindsight from the discovery pass:

- Future onboarding should start from `dot macos audit --json` facts plus the curated discover shortlist, not from a broad recursive scan.
- User-maintained registries live under `state/discover/`: `curated-roots.txt` adds fast roots and `ignore.txt` removes noisy paths by glob or substring.
- `--roots` and `--max-file-size` now make deep/special scans explicit and reproducible.
- Default excludes now cover caches, language-server payloads, package-manager/vendor trees, build outputs, source maps, browser databases, and generated `.app` bundles.
- The interactive text review explains Recommended/Maybe/Risky/Ignored categories and points non-file macOS state at typed audit/module capture flows.

## Goal 1: Capture and reconcile non-file state

**Outcome:** make capture bidirectional where safe and explicit where not safe.

**Scope:**
- Capture Homebrew state from audit facts into a Brewfile-compatible artifact.
- Capture MAS app IDs from audit facts when `mas` is available.
- Capture app bundle IDs/source hints from audit facts for reviewable direct-download checkpoints.
- Capture only the curated defaults surfaced by audit unless the user explicitly extends the curation set; avoid full plist dumps by default.
- Implement subrepo clone-if-missing apply and `dot subrepo status`.
- Integrate secret-backed generated files without committing generated secret material.
- Use the desired artifact paths and absent-artifact semantics from the module contract.
- Plug mutating non-file modules into the existing orchestrator with dry-run, backup/restore or explicit reversibility, apply, verify, fixture, and interrupted-restore test coverage.
- Make scheduled sync behavior safe with new capture modules by preserving clean-repo guards, explainable dry-run output, and `make test-e2e-verify` artifact checks.

**Acceptance:** capture produces reviewable, portable artifacts and labels machine-specific or manual-only state instead of hiding it.

## Suggested sequence

1. **P0:** add capture/apply modules for Brewfile/MAS/apps/defaults/subrepos using the existing audit facts plus orchestrator backup/verify/redaction/restore gates.
2. **P1:** expand the `macos-verification` e2e bundle as each non-file module lands so release readiness remains visible under `state/e2e-runs/<timestamp>/`.
3. **P2:** consider public plugin APIs only after first-party modules pass contract, redaction, fixture, and interrupted-restore tests.

## Non-goals for the next slice

- Do not build a bigger generic filesystem crawler as the primary macOS strategy.
- Do not make Nix mandatory for normal onboarding.
- Do not mutate TCC/privacy databases or Keychain contents.
- Do not expose third-party executable plugins before schemas, redaction, and contract tests are stable.
- Do not apply macOS defaults broadly from full plist exports.
